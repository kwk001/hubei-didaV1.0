# kagent-schema

数据模型 Schema 分析器 - 基于 AI 自动生成数据库 Schema

## 概述

`kagent-schema` 用于分析需求文档，自动生成数据模型和数据库 Schema。它使用 AI 识别实体、字段、关系，并生成对应数据库的 DDL 语句或 ORM Schema，支持 MySQL、PostgreSQL 和 MongoDB。

## 命令模式

### 1. analyze 模式

完整分析模式，生成数据模型文件和 Schema 文件，保存到项目目录。

#### 使用方式

```bash
kagent-schema analyze <input> [选项]
```

### 2. quick 模式

快速分析模式，直接输出 Schema 到控制台，不保存文件。

#### 使用方式

```bash
kagent-schema quick <requirement> [选项]
```

### 3. examples 模式

显示使用示例。

#### 使用方式

```bash
kagent-schema examples
```

kagent-schema analyze "创建一个订单管理系统，包含客户、订单、订单明细"

```bash
kagent-schema analyze ./requirements.md
```

### 需求文档格式

需求文档支持 Markdown 格式，应包含：

**示例：**
```markdown
# 订单管理系统

## 业务实体

### 客户（Customer）
- 客户编号（必填，唯一）
- 客户名称（必填）
- 联系电话
- 联系地址

### 订单（Order）
- 订单编号（必填，主键）
- 客户编号（必填，外键关联客户）
- 订单日期（必填）
- 订单金额
- 订单状态（待处理/已完成/已取消）

### 订单明细（OrderDetail）
- 明细编号（必填，主键）
- 订单编号（必填，外键关联订单）
- 产品编号
- 数量
- 单价
- 小计

## 关系说明

- 一个客户可以有多个订单（1:N）
- 一个订单可以有多个订单明细（1:N）
```

**要点：**
- 明确列出实体名称
- 描述每个字段及其约束
- 说明实体间的关系
- 标注主键、外键、唯一键等

### 字段说明

#### Entity 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 实体名称（PascalCase） |
| `tableName` | string | 表名（snake_case） |
| `displayName` | string | 显示名称（中文） |
| `description` | string | 实体描述 |
| `type` | string | 实体类型（master/detail/dictionary） |
| `fields` | array | 字段列表 |

#### Field 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `name` | string | 字段名（camelCase） |
| `type` | string | 字段类型（String/Number/Boolean/Date/DateTime/JSON/Text/Enum） |
| `length` | number | 长度（可选） |

#### Relationship 字段

| 字段 | 类型 | 说明 |
|------|------|------|

| `toEntity` | string | 目标实体（PascalCase） |
| `foreignKey` | string | 外键字段名 |

### Schema 文件格式

#### MySQL DDL

```sql
-- MySQL DDL
-- Generated by KAgent Schema Analyzer

DROP TABLE IF EXISTS `orders`;
CREATE TABLE `orders` (
  `id` VARCHAR(50) NOT NULL COMMENT '订单ID',
  `customer_id` VARCHAR(50) NOT NULL COMMENT '客户ID',
  `order_date` DATETIME NOT NULL COMMENT '订单日期',
  `amount` DECIMAL(10,2) COMMENT '订单金额',
  PRIMARY KEY (`id`),
  FOREIGN KEY (`customer_id`) REFERENCES `customers`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单主表';
```

#### PostgreSQL DDL

```sql
-- PostgreSQL DDL
-- Generated by KAgent Schema Analyzer

DROP TABLE IF EXISTS "orders" CASCADE;
CREATE TABLE "orders" (
  "id" VARCHAR(50) NOT NULL,
  "customer_id" VARCHAR(50) NOT NULL,
  "order_date" TIMESTAMP NOT NULL,
  "amount" DECIMAL(10,2),
  PRIMARY KEY ("id")
);
COMMENT ON TABLE "orders" IS '订单主表';
COMMENT ON COLUMN "orders"."id" IS '订单ID';
```

#### MongoDB Schema

```javascript
// Mongoose Schema
// Generated by KAgent Schema Analyzer

const mongoose = require('mongoose');
const Schema = mongoose.Schema;

// 订单
const OrderSchema = new Schema({
  customerId: { type: Schema.Types.ObjectId, ref: 'Customer', required: true },
  orderDate: { type: Date, required: true },
  amount: { type: Number },
}, {
  timestamps: true,
  collection: 'orders'
});

module.exports = mongoose.model('Order', OrderSchema);
```

## 工作原理

### DocumentAnalyze 工作流

使用 GenSX Workflow 编排分析流程：

1. **读取需求**：从文本或文件读取需求内容
2. **AI 分析**：调用 LLM 识别实体、字段、关系
3. **生成数据模型**：解析 AI 响应，生成结构化数据模型
4. **生成 ER 图**：使用 Mermaid 格式生成 ER 图
5. **生成 DDL/Schema**：根据数据库类型生成对应的 Schema
6. **保存文件**：写入 JSON 和 Schema 文件

### AI 提示词策略

内置优化的提示词模板，包含：
- 实体识别要点
- 字段类型规范
- 关系类型定义
- Schema 生成规则

### 附加 Prompt 增强

使用 PromptEnhancer 增强基础提示词：
- 附加 prompt 具有最高优先级
- 自动添加强调标记
- 明确要求 AI 优先满足用户要求

## 注意事项

### 需求描述质量

- ✅ 越详细的需求描述，生成效果越好
- ✅ 明确字段类型、约束、关系
- ✅ 提供业务规则和验证要求

### 数据库类型选择

| 数据库 | 适用场景 | 特点 |
|--------|---------|------|
| MySQL | 传统 Web 应用 | 成熟稳定，支持事务 |
| PostgreSQL | 复杂数据类型 | 功能强大，支持 JSON |
| MongoDB | 灵活 Schema | 文档型，适合快速迭代 |

### 实体类型说明

- **master（主表）**：独立的业务实体，如客户、订单、产品
- **detail（明细）**：从属于主表的详细信息，如订单明细、BOM 明细
- **dictionary（字典）**：系统配置类数据，如状态字典、分类字典

### MongoDB 特殊处理

对于 MongoDB：
- **嵌入文档**：明细类实体（detail）会生成为嵌入文档
- **引用关系**：主表间关系使用引用（ObjectId）
- **自动时间戳**：自动添加 timestamps（createTime/updateTime）

### 字段类型映射

#### 通用类型

- `String`：字符串（可指定长度）
- `Text`：长文本
- `Number`：数字（可指定精度和小数位）
- `Boolean`：布尔值
- `Date`：日期
- `DateTime`：日期时间
- `JSON`：JSON 对象
- `Enum`：枚举（需提供 enumValues）

#### 数据库特定映射

**MySQL：**
- String → VARCHAR
- Number → INT / DECIMAL
- Boolean → TINYINT(1)
- Date → DATE
- DateTime → DATETIME
- JSON → JSON
- Enum → ENUM

**PostgreSQL：**
- String → VARCHAR
- Number → INTEGER / DECIMAL
- Boolean → BOOLEAN
- Date → DATE
- DateTime → TIMESTAMP
- JSON → JSONB
- Enum → VARCHAR

**MongoDB：**
- String → String
- Number → Number
- Boolean → Boolean
- Date/DateTime → Date
- JSON → Schema.Types.Mixed
- Enum → String (with enum validation)

## 常见问题

### Q: 生成的 Schema 不符合预期怎么办？

A:
1. 检查需求描述是否足够详细
2. 使用 `--prompt` 添加额外要求
3. 调整 `--temperature` 参数（降低温度提高确定性）
4. 查看 `data-model.json` 确认数据模型是否正确

### Q: 如何指定特殊的字段约束？

A: 在需求文档中明确描述：
```markdown
- 手机号（必填，11位数字，唯一）
- 邮箱（可选，符合邮箱格式，唯一）
- 状态（必填，枚举：待审核/已通过/已拒绝）
```

### Q: 支持多对多关系吗？

A: 支持。AI 会自动识别多对多关系并生成中间表（MySQL/PostgreSQL）或适当的引用结构（MongoDB）。

### Q: 可以自定义表名吗？

A: AI 会自动生成表名（snake_case）。如需自定义，可以在需求中明确说明：
```markdown
### 订单（表名：t_order）
```

### Q: 生成的 Schema 可以直接使用吗？

A: 生成的 Schema 是一个良好的起点，但建议：
1. 人工审核数据模型
2. 根据实际业务调整
3. 添加索引优化
4. 补充业务约束
